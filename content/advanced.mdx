# 고급 설정

## CLAUDE.md

`CLAUDE.md`는 에이전트 동작 원칙을 고정하는 핵심 설정 파일입니다.

- **전역 `CLAUDE.md`** (`~/.claude/CLAUDE.md`): 모든 프로젝트에 공통으로 적용되는 개인 규칙
- **프로젝트 `CLAUDE.md`** (`./CLAUDE.md`): 저장소 루트에 두고 팀 규칙과 구체적 범위 정의
- 충돌 시에는 프로젝트 규칙이 우선입니다.

### 프로젝트 CLAUDE.md 템플릿

아래를 복사해 프로젝트 루트의 `CLAUDE.md`에 저장하고 필요에 맞춰 수정하세요.

```markdown
# 프로젝트 규칙

## 기술 스택
- 언어: Python 3.11
- 프레임워크: FastAPI
- 데이터베이스: PostgreSQL
- 테스트: pytest

## 작업 범위
- 수정 가능: /src, /tests
- 금지 영역: /config (보안 파일), /docs (따로 리뷰)
- 의존성 추가: 팀 리드 승인 필수

## 작업 규칙
- 커밋 전: `pytest -v` 반드시 실행
- 함수 주석: 한국어로 작성 (docstring 포함)
- 기능 추가: 테스트 함께 제출

## 금지 사항
- 외부 라이브러리 무단 추가 금지
- 직접 SQL 작성 금지 (ORM 사용)
- 시크릿 키를 코드에 하드코딩 금지
```

<Callout type="tip" title="팁">
전역에는 개인 워크플로우만 두고, 프로젝트에는 팀이 따라야 할 파일 경로/금지 영역/테스트 명령처럼 실행 가능한 규칙만 넣으면 충돌을 줄일 수 있습니다.
</Callout>

## Hooks

Hooks는 특정 시점에 자동으로 검증이나 후처리를 붙일 수 있는 실행 지점입니다.

- `SessionStart`: 세션 시작 시 환경 점검, 브랜치 확인, TODO 로드
- `PreToolUse`: 명령 실행 전 위험 커맨드 차단, 인자 검증
- `PostToolUse`: 결과 로그 수집, 실패 패턴 감지, 후속 알림
- `Stop`: 세션 종료 전 요약 저장, 임시 산출물 정리

### 설정 예제

`~/.claude/settings.json`에서 Hooks를 정의합니다.

**예제 1: 파일 편집 후 자동 git add**

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "tools": ["Edit", "Write"],
        "action": "run",
        "command": "cd {cwd} && git add {affectedFiles}"
      }
    ]
  }
}
```

**예제 2: 위험한 Bash 명령 차단**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "tools": ["Bash"],
        "blockedPatterns": [
          "git reset --hard",
          "rm -rf",
          "sudo"
        ],
        "action": "warn"
      }
    ]
  }
}
```

자동화는 생산성을 높이지만, 차단형 Hook은 오탐 시 작업을 멈출 수 있으니 작은 규칙부터 점진적으로 적용하세요.

<Callout type="tip" title="팁">
`PreToolUse`에는 보안성 규칙(예: 위험한 커맨드 차단)을, `PostToolUse`에는 관측성 규칙(로그/자동 커밋)을 배치하면 운영이 단순해집니다.
</Callout>

## Skills

Skills는 재사용 가능한 작업 절차를 묶어 슬래시 명령으로 빠르게 실행하는 방식입니다.

- 팀에서 반복되는 작업을 Skill로 표준화하면 품질 편차를 줄일 수 있습니다.
- 명령어는 짧고 의미가 명확해야 합니다.

예시:

```bash
/review
/ship
```

- `/review`: 코드 리뷰 체크리스트(버그, 보안, 테스트 누락) 기반 점검
- `/ship`: 릴리스 준비(빌드, 테스트, 변경 요약, 배포 체크) 자동 실행

<Callout type="tip" title="팁">
Skill은 "입력 형식 + 실행 단계 + 완료 조건" 3요소를 고정하면 팀원 간 결과가 일관됩니다.
</Callout>

## Sub-agents

Sub-agent는 메인 에이전트가 작업을 분할 위임할 때 쓰는 독립 실행 단위입니다.

- 격리 컨텍스트: 부모 문맥을 필요한 만큼만 전달해 간섭을 줄입니다.
- 내장 타입: `explorer`, `worker`, `reviewer` 등 역할별 특화 에이전트를 선택합니다.
- `Agent()` 위임: 탐색/구현/검증을 병렬로 분리해 처리 시간을 단축합니다.

Sub-agent를 늘릴수록 조율 비용이 커지므로, 의존성이 낮은 작업부터 분리하는 것이 효율적입니다.

<Callout type="tip" title="팁">
먼저 "파일 소유 범위"를 나눈 뒤 위임하면 충돌을 크게 줄일 수 있습니다.
</Callout>

## Permissions

Permissions는 도구 실행 권한을 단계적으로 통제해 오작동과 권한 오남용을 방지합니다.

| 모드 | 설명 | 사용 시기 |
|---|---|---|
| `default` | 모든 작업 실행 시 확인 | 초보자, 신규 프로젝트 |
| `acceptEdits` | 파일 편집만 자동 승인, Bash는 확인 | 작업에 집중하되 Bash만 검토 |
| `bypassPermissions` | 모든 도구 자동 실행, 확인 없음 | 신뢰된 작업 (배포, 테스트) |
| `plan` | 계획만 세우고 실행은 직접 승인 | 복잡한 작업의 방향 검토 |

<Callout type="warn">
`bypassPermissions`는 신뢰할 수 있는 작업에만 사용하세요. 실수로 중요 파일을 삭제할 수 있습니다.
</Callout>

### 권한 전환 예시

```bash
# 현재 권한 확인
/permissions show

# 편집만 자동 승인
/permissions set acceptEdits

# 전체 자동 실행 (주의!)
/permissions set bypassPermissions

# 기본값으로 복원
/permissions set default
```

<Callout type="tip" title="팁">
기본 정책은 `default`로 두고, 반복된 작업이 명확할 때만 `acceptEdits`로 전환하세요.
</Callout>
